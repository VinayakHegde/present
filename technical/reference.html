Technical Reference

Consolidated revision notes — frontend, backend, AI engineering, infrastructure, security — plus a spoken-answer drill. Skim top to bottom in ~30 minutes.

1Frontend — React & TypeScript

React + TypeScript essentials

  • Component composition — reusable building blocks over one-off screens (a common live-coding focus)
  • React Query for server state: cache keys, staleTime, keepPreviousData, invalidation, optimistic updates; UI state stays local
  • Type boundaries: API types generated from OpenAPI; never any at the edges

Real-time / WebSockets

  • Reconnect with backoff; resync state on reconnect
  • Don't block UI on every tick — batch/throttle renders
  • Show connection status and a freshness timestamp; degrade gracefully to polling
  • Never show stale data as live — finance users notice immediately

Testing

  • RTL for behaviour, not implementation; MSW for API mocks
  • 4-layer test parity: calc → BFF → hook → component
  • Golden fixtures for financial correctness paths

React vs Vue — if probed

  • React: hooks, React Query for server state, explicit re-render model
  • Vue: reactivity system, SFCs — different syntax, same mental model for product engineers
  • Don't pretend production experience you lack — say confident ramp on modern TS frameworks

2Backend & APIs

FastAPI, Pydantic & Postgres

  • Pydantic v2 validation at the boundary; async SQLAlchemy underneath
  • OpenAPI as the contract; generated clients keep FE types honest
  • Materialised snapshots vs live aggregation — know when each wins

Auth & permissions (BFF pattern)

  • Browser talks to the BFF only; BFF attaches user identity; domain services are never public
  • Metric/data visibility via an auth service + per-user scope — calculations respect entity-level permissions
  • Financial-data products make access control explicit business logic — expect RBAC / entity-level permission questions
"Auth lives at the BFF boundary. The browser never calls domain services directly. The BFF validates the session, attaches identity, and enforces scope before shaping the response — the same pattern applies to credit-data visibility in a financial-data SaaS."

Event-driven patterns

  • Async materialisation: recompute snapshots off the hot path via queues
  • Invalidation on definition change or source refresh — not time-based guessing
  • Idempotent workers: key work by natural identity (e.g. tenantId + definitionId + scopeHash + sourceWatermark), upsert, dedupe-lock

3AI Engineering — MCP, RAG, agents

MCP (Model Context Protocol)

"MCP is a way to give AI agents structured access to tools, systems and data sources through a common protocol. The important part is not just connecting tools, but controlling access, permissions, auditing, and making sure the agent only has the context and actions it should have."
  • Shape: client (agent) ↔ MCP server ↔ tools/resources. JSON-RPC. Tool schemas define inputs; resources give read-only context. Stdio vs SSE transport
  • Design: narrow tools (searchDocs, getTicket) beat mega-prompts and raw DB access; separate read from write
  • In practice: one MCP server per integration; auth passthrough so the agent acts as the user; audit logs on every tool call
  • Hands-on angle: daily MCP use inside an AI IDE (issue trackers, browser tools) — client/server model and tool schemas from experience, not theory

RAG & knowledge pipelines

Headline architecture: ingest → chunk → embed → store (pgvector / Pinecone) → retrieve with metadata filters → prompt with citations → eval.

  • Chunking: by doc section / ticket / PR — preserve metadata (source, author, date, ACL)
  • Embeddings: managed (e.g. Bedrock) or dedicated model — same model for index and query; re-embed on doc change via webhook pipeline
  • Retrieval: hybrid search (semantic + keyword) for jargon-heavy engineering docs
  • Context assembly: top-k with diversity; token budget; cite sources in the UI
  • Fail gracefully when retrieval comes back empty — "I don't know" over inventing

Evals

  • Golden question sets per domain, with expected citations
  • Regression run on any change to chunk size, model, prompt, or index
  • Track hallucination rate and "no answer" rate — both matter
  • Measure retrieval precision/recall separately from generation quality

Integrations & connectors — design principles

  • One connector service per domain — don't build a god-service
  • MCP exposes curated tools; the connector handles auth, pagination, caching
  • Normalised connector interface (search, get, list) with MCP tools on top
  • Sync jobs feed the index; live API for fresh single-record lookups
  • Event webhooks (repo push, ticket update) → queue → index update — don't poll everything

Workflow automation: use workflow-automation tools (e.g. n8n) as orchestration for deterministic multi-step workflows — ticket update → chat notify → index refresh — while the AI/agent layer handles the ambiguous reasoning steps. Don't make an LLM do what a workflow engine does reliably.

4Infrastructure — Kubernetes, containers, cloud

Kubernetes, containers & AWS — talking points

  • Health/readiness probes; graceful shutdown for in-flight LLM requests
  • Horizontal scale for stateless APIs; separate workers for embedding/index jobs; autoscale inference carefully
  • Observability: latency p95, error rate, token cost per request, tool-call failures
  • Mapping from ECS experience: task definition → Deployment · service discovery → K8s Service · ALB → Ingress
  • Know when serverless (Lambda) vs containerised (ECS) — burst/event-shaped work vs steady services with warm state

Honest gap framing: if day-to-day K8s YAML is lighter, say the concepts and production discipline are solid and the ramp is quick — internal tools still handle sensitive data, so the same rigour applies.

5Security & governance

Practical controls (regulated environments)

  • Data classification on every connector; block restricted classes (PII, financials, credentials) from LLM paths
  • Secrets never in prompts or logs; vault + short-lived tokens
  • Model provider choice (e.g. Bedrock) for enterprise data residency / DPA
  • Audit trail: who asked what, which tools fired, which docs were retrieved
  • Data minimisation: smallest context window needed; redact PII/secrets before the model call
  • Output filtering: block patterns (API keys, account numbers); scan outbound responses
  • Human-in-the-loop for writes and destructive actions; safe-default templates so non-engineers can't install arbitrary tools
"The agent should only see what the user is allowed to see, through narrow tools — not a firehose of company data in the system prompt."

7Quick Q&A drill

Spoken-answer practice. Structure: headline → how → result/metric, under 2 minutes each.

Walk me through data flow end-to-end

Warehouse facts + Postgres definitions → calc service applies rules → validation → FastAPI → BFF shapes the response → React/export. The browser never hits the calc service. Invalidation on definition change or source refresh.

Why server-side calculation?

Single source of truth. Client-side duplicates logic and drifts — it's a trust incident if UI and export disagree.

How do you guarantee correctness?

Golden fixtures from real customer models; validation at the calculation boundary; CI on calc paths; visible failure over silent wrong numbers.

Caching vs live calculation?

Two tiers, not either/or. Pinned metrics: async snapshots (~120ms reads). Ad-hoc: live calc + React Query dedupe. A lastCalculatedAt field makes staleness explicit.

How does it scale 10×?

Hot path scales with snapshot reads. Cold path needs warehouse indexing + queue-backed recompute. The architecture already separates the tiers.

How do you handle idempotency?

Materialisation keyed by (tenantId, definitionId, scopeHash, sourceWatermark). Upserts are idempotent; a dedupe lock guards the async worker.

How do you approach frontend architecture?

Start from user journeys and data boundaries. Server state in React Query; UI state local. Shared components for repeated patterns (tables, export). Contract-first APIs so FE types stay honest. Feature flags for safe rollout.

How do you handle real-time data in the UI?

Separate transport (WebSocket/poll) from presentation. Throttle renders; show a freshness timestamp; handle disconnect explicitly. Never show stale data as live — finance users notice immediately.

React vs Vue — preference?

React is my primary stack and I'm productive day one. Vue is familiar in concept — I'd ramp quickly. I care more about product outcomes and team conventions than framework religion.

How do you balance speed and quality?

Scope MVPs ruthlessly with product. Golden fixtures + tests on correctness paths. Ship behind flags; measure adoption. Proof point: one week discovery-to-prod on a metrics platform with zero P1s — speed with guardrails.

How do you use AI in your workflow?

Accelerator for scaffolding, tests, mocks — never architecture or financial rules. Full CI on everything; no customer data in prompts; I own every merge.

Mentoring example?

Built a 4-layer test template with a junior FE; they shipped IRR + export tests independently; the checklist now lives in the squad PR template.

Biggest near-miss?

Nearly shipped client-side calculation — killed it after a spike showed Excel-parity failure on an FX month-end edge case.

What would you do differently?

Instrument adoption/export analytics in the PRD from day one (we backfilled post-GA). Run the data-platform warehouse spike in discovery, not mid-build.
