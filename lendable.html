Likely Q&A

Structure: headline → how you did it → result/metric → what you'd do at Lendable. Keep under 2 min each.

① Tell me about an AI tool or workflow you have built or introduced

Primary story — AI-assisted delivery at Lantern:

  • Problem: One-week push to ship calculated metrics; test/fixture work was the bottleneck
  • What I introduced: Cursor + spec-first AIAD workflow — AI for test scaffolding, MSW handlers, BFF boilerplate; strict guardrails on what AI may touch
  • Guardrails: Rules, validation logic, architecture never via AI; every line reviewed; 4-layer test parity required per contract change
  • Result: ~50% faster fixture/mock work; 50+ tests green throughout; discovery → production in one week; zero P1 at GA

Secondary — squad adoption: Shared PR checklist, test templates, patterns junior engineers reused on export API + widgets.

Frame as: internal tooling mindset — make engineers faster with guardrails, not magic.

② How would you expose company knowledge safely to an AI agent?
  1. Classify data — public internal docs vs restricted (PII, financials, credentials)
  2. Retrieve, don't dump — RAG with scoped indexes per domain/team; embeddings + metadata filters
  3. Permission-aware retrieval — agent queries as the authenticated user; enforce ACLs at source (GitHub, Jira, Drive)
  4. MCP tool boundaries — narrow tools (searchConfluence, getTicket) not raw DB access; read vs write separated
  5. Audit & logging — what was retrieved, which tools called, prompt/response retention policy
  6. Human-in-the-loop for writes (create ticket, merge PR) and sensitive reads
"The agent should only see what the user is allowed to see, through narrow tools — not a firehose of company data in the system prompt."
③ How would you measure whether an AI tool improves developer productivity?

Define metrics before shipping; baseline first:

  • Adoption: DAU/WAU, % engineers using pronto weekly, retention after first use
  • Time saved: survey + sample task timing (e.g. test scaffolding, incident triage, boilerplate)
  • Flow metrics: PR cycle time, lead time for changes, deployment frequency (DORA)
  • Quality signals: defect/rework rate, CI failure rate, incidents attributed to AI-generated code
  • Qualitative: engineer satisfaction, "would you recommend" — especially from skeptics

See Measure success section for the full list to cite.

④ How would you design integrations for Slack, Jira, GitHub, Google Workspace and Snowflake?

Common layer:

  • OAuth/service accounts per system; secrets in vault; rotate tokens
  • Normalised connector interface: search, get, list — MCP tools on top
  • Rate limiting, caching, incremental sync where possible
  • Event webhooks (GitHub push, Jira update) → queue → index update, not poll-everything

⑤ How do you manage hallucination, permissions and data leakage risks?
  • Grounding: RAG with citations; require retrieved context for factual claims; "I don't know" over inventing
  • Permissions: enforce at retrieval + tool call; no elevation without explicit admin tooling
  • Data minimisation: smallest context window needed; redact PII/secrets before model call
  • Output filtering: block patterns (API keys, account numbers); scan outbound responses
  • Evals: golden Q&A sets per domain; regression when prompt/model/index changes
  • Human review: mandatory for destructive actions and customer-facing output
  • Monitoring: log tool usage, flag anomalous broad queries, alert on policy violations
⑥ What is your experience with Kubernetes and containerised deployment?
"I've shipped containerised services on AWS — Docker images, CI-built artifacts, environment-specific config, health checks, and observability. I understand Kubernetes as the orchestration layer on top of that: deployments, services, ingress, secrets, resource limits, and rollbacks. For internal AI tooling I'd containerise the MCP servers and API gateways, use K8s for consistent deploys across envs, and wire in metrics/tracing from day one — same discipline as any production service, because internal tools still handle sensitive data."
  • Concrete: Docker + ECS at Lantern; maps to Pods/Deployments/Services mentally
  • AI-specific: separate services for embedding/index jobs vs request-serving API; autoscale inference carefully
  • Honest gap: if K8s YAML day-to-day is lighter, say productive quickly — concepts and production discipline are solid
⑦ How would you decide what to build first in the AI engineering team?
  1. Talk to engineers — shadow workflows; find repeated manual steps and trust gaps
  2. Quantify pain — frequency × time × number of people; focus on high-leverage bottlenecks
  3. Feasibility & risk — quick win with read-only integrations before write actions
  4. Align with pronto roadmap — extend what's already adopted vs parallel tools
  5. Measure from day one — baseline before launch; 4-week checkpoint
  6. Say no to demos that don't ship — one tool used weekly beats five POCs

MCP (if asked)

Simple answer (memorise)

  • At Lendable: MCP servers per integration; read/write separation; auth passthrough; audit logs

RAG & knowledge pipelines

Headline architecture

Ingest → chunk → embed → store (pgvector / Pinecone) → retrieve with metadata filters → prompt with citations → eval.

  • Chunking: by doc section / ticket / PR — preserve metadata (source, author, date, ACL)
  • Embeddings: Bedrock or dedicated model; re-embed on doc change via webhook pipeline
  • Retrieval: hybrid search (semantic + keyword) for jargon-heavy engineering docs
  • Context assembly: top-k with diversity; token budget; cite sources in UI
Evals (mention confidently)
  • Golden questions per domain with expected citations
  • Regression when changing chunk size, model, or index
  • Track hallucination rate and "no answer" rate — both matter

Integrations (reference)

Design principles
  • One connector service per domain — don't build a god-service
  • MCP exposes curated tools; connector handles auth, pagination, caching
  • Sync jobs for index; live API for fresh single-record lookups
  • Security review before each new data source — data classification checklist
n8n / workflow automation angle

Lendable uses n8n for internal automation. Frame as: orchestration for multi-step workflows (e.g. Jira ticket → Slack notify → index update) while AI/agent layer handles ambiguous reasoning steps.

Security & governance

Practical controls (Lendable is regulated fintech)
  • Data classification on every connector; block restricted classes from LLM paths
  • Secrets never in prompts or logs; vault + short-lived tokens
  • Model provider choice (Bedrock) for enterprise data residency / DPA
  • Audit trail: who asked what, what tools fired, what docs retrieved
  • Templates + guardrails for non-engineers — safe defaults, no arbitrary MCP installs
  • Partner with security early — not a gate at the end

Kubernetes & containers

Talking points
  • Docker images built in CI; immutable tags; scan for vulnerabilities
  • Config via env/secrets; 12-factor services for MCP servers and APIs
  • Health/readiness probes; graceful shutdown for in-flight LLM requests
  • Horizontal scale for stateless API; separate workers for embedding jobs
  • Observability: latency p95, error rate, token cost per request, tool call failures
  • Map from ECS experience: task definition → Deployment, service discovery → K8s Service, ALB → Ingress

Revise (30 min skim)

MCP

Client (agent) ↔ MCP server ↔ tools/resources. JSON-RPC. Tool schemas define inputs. Resources for read-only context. Stdio vs SSE transport.

RAG

Retrieve relevant chunks → augment prompt → generate with citations. Fail gracefully when retrieval empty.

Embeddings

Vector representation of text for semantic similarity. Same model for index and query. Metadata filters critical for ACLs.

Evals

Golden sets, human review sample, automated citation checks, regression on model/index changes. Measure precision/recall of retrieval separately from generation quality.
