3 · Scoping questions

Questions to answer before building — a few upfront, not endless requirements gathering.

Product questions

  • Who is the primary reviewer: engineers, technical writers, design system maintainers, product teams?
  • Should the system create drafts or directly publish updates?
  • What is worse initially: missing a doc update or suggesting irrelevant updates?
  • What kind of documentation drift hurts users most today?
  • Are suggestions expected per PR, per release, or on demand?

Technical questions

  • Which code host should we assume first: GitHub, GitLab or Bitbucket?
  • What capabilities does the CMS API provide: read, draft, update, publish, version history?
  • How is documentation structured: pages, components, APIs, packages, product areas?
  • Is there ownership metadata for docs or code areas?
  • Do we have access to PR descriptions, changed files and commit metadata?

Success questions

  • What would MVP success look like?
  • What user behaviour proves trust?
  • Is precision or recall more important at first?
  • What failure would damage trust most?
★ Sharpest scoping question: “Is it worse for the MVP to miss some documentation updates, or to create too many irrelevant suggestions?”

6 · Architecture overview

Product scope and success criteria

I would scope this as a documentation suggestion and review platform, not an automatic documentation generation system. The goal is not to publish AI-written docs directly, but to detect code changes that may require documentation updates, generate evidence-backed suggestions, and let users review, edit, accept, or reject them.

Trust is central, so every suggestion should explain:

  • Why it was generated
  • What code change triggered it
  • Which document it affects
  • What confidence level the system has
The key design choice is to keep webhook ingestion lightweight and push analysis into asynchronous jobs. The system should create explainable, reviewable documentation suggestions rather than directly publishing AI-generated changes.
Avoid heavy processing inside the webhook. The ingestion API should validate and store the event quickly, then hand off processing to a background job.

Five-layer design

Inside the product-owned core, I would design this as five layers.

  1. Ingestion — receives PR merged events from GitHub or GitLab, validates the webhook, stores a durable ChangeEvent, deduplicates it, and queues asynchronous processing.
  2. Impact analysis and document discovery — fetches the PR diff, classifies whether the change is user-facing, extracts entities like API endpoints or config keys, and retrieves likely affected CMS documents using mappings, search, semantic retrieval, and past feedback.
  3. Suggestion engine — generates a targeted documentation patch, not a full rewrite, and includes evidence, explanation, and confidence.
  4. Human review — users can accept, edit, or reject the suggestion. Accepted suggestions create a CMS draft or update through the CMS API.
  5. Learning and evaluation — store review outcomes and use them to improve ranking, confidence, style, and future suggestions.

Human-in-the-loop by design

To build trust, this is a human-in-the-loop suggestion system. When a PR is merged, we analyze the code diff, classify whether the change is documentation-impacting, retrieve the relevant CMS documents, and generate a proposed edit.

The user sees the proposed diff, the affected documentation page, the code change that triggered it, an explanation, and a confidence score. The system does not auto-publish. The user can accept, edit, or reject the suggestion, and that action becomes feedback for future suggestions.

In the ingestion layer, I would store the raw webhook payload immutably for audit and replay. The synchronous webhook handler should only do lightweight work: validate the signature, check that the event is a merged PR, verify required fields, deduplicate using the webhook delivery ID or merge commit SHA, create a normalized ChangeEvent, enqueue a background job, and return quickly.

I would not fetch large diffs, call the CMS, or invoke AI in the request path. Background workers would then fetch the canonical PR data and diff from the code host, run impact analysis, retrieve relevant docs, generate suggestions, and handle retries for timeouts, rate limits, or transient failures.

In the impact analysis layer, I would first decide whether the PR has documentation impact at all. We should classify the change using signals like changed file paths, public API changes, OpenAPI or GraphQL schema diffs, config keys, SDK methods, CLI commands, tests, PR title, and code ownership. Internal refactors or test-only changes should usually stop there.

If the change appears user-facing, we extract structured entities such as endpoints, parameters, error codes, or feature names. Then we query a local indexed mirror of the CMS, where documents are chunked by section and enriched with metadata like title, slug, tags, owner, product area, and version.

For retrieval, I would use a hybrid approach: deterministic code-to-doc mappings, keyword search, metadata filtering, semantic search, and historical feedback from accepted or rejected suggestions. The output should be a ranked list of candidate document sections with evidence and confidence, not just a page ID.

After impact analysis, the suggestion engine receives a grounded context package: the PR summary, relevant diff snippets, extracted entities like API endpoints or config keys, the current CMS document section, its version, and any team style guidelines. I would use the LLM to generate a structured, patch-style suggestion rather than a full rewrite. The output should include the proposed diff, a plain-English explanation, evidence from the PR, assumptions, and uncertainty.

Before storing it, I would validate that the output matches the schema, that the patch applies to the same CMS version, and that the suggestion is scoped to the intended section. The suggestion is stored as pending and shown in a review UI with before/after comparison, confidence reasons, and source evidence.

I would not rely purely on the LLM’s confidence; the system confidence should combine retrieval quality, exact code-to-doc matches, validation results, and historical feedback.

In the review layer, the user should see a diff-based review screen rather than a black-box generated document. The screen should show the current CMS section, the proposed patch, a rendered preview, the PR that triggered the suggestion, evidence from the diff, rationale, assumptions, and confidence reasons. The user can accept, edit, or reject each suggestion.

If they edit, we store both the original AI output and the final accepted version because that becomes valuable feedback. If they reject, we capture a structured reason such as wrong page, not user-facing, already documented, or incorrect. On accept, I would usually create a CMS draft or move it into a publishing queue rather than auto-publish.

Before applying anything, the backend checks permissions and verifies that the CMS document version has not changed since the suggestion was generated. If it has changed, we mark the suggestion stale or regenerate it. This keeps the workflow safe and preserves user trust.

In the learning layer, every review action becomes structured feedback. We store whether the suggestion was accepted as-is, edited and accepted, or rejected, along with rejection reasons, the original AI output, the final accepted version, the retrieved document sections, confidence score, prompt version, model version, CMS document version, and enough context to replay what the system knew at generation time.

I would treat this as a product evaluation loop, not automatic model self-improvement. Rejection reason distribution tells us which part of the pipeline is failing. If users reject suggestions as not user-facing, the impact classifier needs work. If they reject them as wrong document, retrieval or ranking is failing. If they edit before accepting, retrieval was probably right, but generation, tone, or completeness needs improvement.

Over time, we can use this feedback to tune retrieval thresholds, refine code-to-doc mappings, improve prompt examples and style guidance, calibrate confidence against real acceptance rates, and build explicit evaluation sets before changing prompts or models. Key metrics would include acceptance rate, edit rate, rejection reason mix, wrong-document rate, time-to-review, and suggestion-to-publication rate. The goal is that users experience fewer irrelevant suggestions, faster review, and higher trust — not hidden behavior changes they cannot understand.

12 · Component reference

Responsibilities and design notes for every box in the architecture diagram.

Layer 1 · External systems

Code Repository

Source of truth for code changes. Emits webhook events when pull requests merge or releases ship.

Responsibilities

  • Hosts application and library source code
  • Emits PR merged / push / release webhooks
  • Provides diff, changed files, PR title and description via API

Details

  • MVP: pick one provider first, likely GitHub
  • Product validates webhook signatures and stores raw payloads — it does not own the repo
External responsibility: emit events. Product responsibility: decide which events matter and process them.
Headless CMS

Third-party system where developer documentation already lives. Exposes a public API for read and draft write.

Responsibilities

  • Stores published and draft documentation pages
  • Provides CMS API for read, draft create/update, metadata
  • May expose version history and ownership fields

Details

  • Product indexes content for search — does not replace the CMS
  • Accepted suggestions become CMS drafts, not direct publishes in MVP
Clarify early: which CMS capabilities exist (draft only vs publish, versioning, webhooks for content changes).
AI Provider

External LLM API used for summarisation and drafting. Helpful but not the owner of the workflow.

Responsibilities

  • Summarise code diffs and PR context
  • Draft natural-language documentation updates
  • Never decides what gets published

Details

  • Ground prompts with retrieved doc content and change evidence
  • Redact secrets; respect data-handling policies
  • Start with general models — fine-tuning is not MVP
AI is part of the workflow, not the owner of the workflow.
Users / Reviewers

Engineers, technical writers, or maintainers who judge whether a suggestion is correct before anything ships.

Responsibilities

  • Review suggested diffs with evidence
  • Accept, edit, or reject suggestions
  • Provide rejection reasons that improve future quality

Details

  • UX must make review faster than manual doc hunting
  • Trust comes from control — not from confident AI language
Identify the primary reviewer early — it shapes noise tolerance and UI design.

Layer 2 · Product-owned components

Repository Event

Normalised signal that something meaningful changed in code — typically a merged PR or release.

Responsibilities

  • Capture repo id, PR id, merge commit SHA
  • Trigger downstream processing after ingestion
  • Use idempotency: repo + PR + commit to avoid duplicates

Details

  • Not every commit needs a doc suggestion — classifier filters noise later
  • Release events and reconciliation scans can produce events too
The trigger is cheap; the expensive work happens async.
Change Ingestion API

Webhook endpoint that validates incoming repository events and persists them quickly.

Responsibilities

  • Validate webhook signature
  • Store raw event payload
  • Return 2xx fast — no heavy analysis here
  • Use idempotency key: repo_id + pr_id + merge_commit_sha

Details

  • Duplicate webhooks should not create duplicate jobs
  • If ingestion is slow, the code host may retry and amplify load
Key design choice: lightweight ingestion, async processing.
Event Store

Durable log of raw repository events for replay, audit, and debugging.

Responsibilities

  • Keep immutable raw webhook payload
  • Enable reprocessing after bug fixes
  • Support admin/debug views

Details

  • Helps when webhooks are lost — pairs with scheduled reconciliation
  • Useful for tracing why a suggestion was or was not created
Replay is a core part of failure recovery.
Job Queue

Async backbone that runs classification, retrieval, and suggestion generation outside the webhook request.

Responsibilities

  • Process events in background workers
  • Retry transient failures (CMS down, LLM timeout)
  • Dead-letter queue for poison or repeated failures
  • Support backpressure when many PRs merge at once

Details

  • Implementation: Inngest, SQS, BullMQ, Trigger.dev, etc.
  • Observability on job latency and failure rate
The async backbone is what keeps ingestion safe under load.
Change Classifier

Understands what changed in the codebase well enough to scope documentation impact.

Responsibilities

  • Read changed files, PR title, description, commit messages
  • Detect affected APIs, packages, components, or public surfaces
  • Summarise change for retrieval and LLM context
  • Skip or downgrade noisy changes (formatting-only, deps-only)

Details

  • Large PRs: chunk diff, summarise, prioritise high-impact paths
  • Output feeds relevance engine and suggestion generator
Combine rules/metadata with LLM summary — do not rely on LLM alone to read entire diffs.
Documentation Content Index

Local searchable copy of CMS documentation, synced on a schedule or via CMS webhooks if available.

Responsibilities

  • Sync CMS pages, chunks, tags, ownership metadata
  • Track last updated timestamps
  • Power fast search without hitting CMS on every PR
  • Handle freshness — re-sync when docs change

Details

  • Start with Postgres full-text search on indexed content
  • Store page ids needed for draft updates via CMS API
Trade-off: indexed copy vs live CMS query — index wins speed; must stay fresh.
Relevance Engine

Finds which documentation pages are likely stale or affected by a code change.

Responsibilities

  • Match classifier output to indexed docs
  • Use metadata, keyword search, code-to-doc mappings
  • Rank candidates; prefer precision over recall in MVP
  • Optionally use embeddings later — not required for v1

Details

  • Poor retrieval → irrelevant suggestions → users lose trust
  • Show why each page was selected in the review UI
  • Allow manual linking when nothing matches
Key scoping question — miss updates vs noisy suggestions — shapes the ranking threshold here.
Suggestion Generator

Creates a proposed documentation update grounded in code change and existing doc content.

Responsibilities

  • Combine diff summary + retrieved docs + style guidance
  • Produce suggested patch, rationale, and evidence file links
  • Assign confidence level
  • Never publish directly — output goes to suggestion store

Details

  • Ground in evidence to reduce hallucination
  • Human review remains mandatory for MVP
  • Log system version for evaluation over time
AI generates; product owns prompt design, grounding, and guardrails.
Suggestion Store

System of record for each proposed doc update and its review lifecycle.

Responsibilities

  • Store pending / accepted / edited / rejected status
  • Persist suggested patch, rationale, evidence, confidence
  • Link to repo, PR, merge commit, CMS page id
  • Track reviewer and reviewed_at timestamps

Details

  • Enables review UI and audit trail
  • system_version field supports before/after evaluation of prompt changes
This is where product state lives — distinct from CMS content until accepted.
Review UI

Human control point: compare current doc, suggested change, evidence, and take action.

Responsibilities

  • Show PR link, changed files, affected doc page
  • Side-by-side or diff view of current vs suggested content
  • Display rationale, evidence, confidence
  • Actions: accept, edit, reject, ignore path
  • Capture rejection reason dropdown + free text

Details

  • Accessibility: keyboard nav, labels, not colour-only status
  • Principle: make the reviewer's job easier, not harder
The trust boundary lives here — this is what users actually experience.
CMS Draft Update

Writes an accepted (or edited) suggestion into the headless CMS as a draft only.

Responsibilities

  • Call CMS API to create/update draft
  • Respect CMS permissions for the reviewer
  • Preserve version history if CMS supports it
  • Never auto-publish in MVP

Details

  • If CMS is unavailable: retry queue, show user-visible delay
  • Audit log records who accepted what and when
Draft-first publishing is a deliberate trust decision, not a technical limitation.
Feedback + Evaluation Store

Captures human decisions and feeds improvement loops — not magic model self-learning.

Responsibilities

  • Store accept / edit / reject with reasons
  • Track edited patches vs original suggestion
  • Build evaluation sets from real examples
  • Improve prompts, retrieval ranking, and thresholds over time

Details

  • Rejection reason distribution is a key product metric
  • Phase 3+ evolution: explicit eval sets and ranking tuning
We improve retrieval and prompts deliberately from feedback — not fine-tune on day one.

Layer 3 · Trust boundary steps

AI-generated suggestion

Output of the suggestion generator — useful draft, not authoritative truth.

Responsibilities

  • Must include evidence and confidence
  • Stays in suggestion store until reviewed
  • Can be wrong — system assumes that

Details

  • Low-confidence suggestions should look uncertain, not oversold
  • No silent writes to CMS
Assistive output entering the trust boundary.
Human review

Mandatory checkpoint before any documentation changes reach the CMS.

Responsibilities

  • Reviewer verifies accuracy against PR and product behaviour
  • Can edit before accept
  • Rejection captures why the system was wrong

Details

  • Acceptance rate and edit-before-accept rate measure real value
  • Without this step, stale or wrong docs damage user trust faster than missing docs
Core principle: trust first, automation later.
Draft update

Accepted content written to CMS as draft — visible in CMS workflow, not live docs yet.

Responsibilities

  • Separates AI-assisted draft from published documentation
  • Allows existing CMS review/publish flows to remain
  • Supports rollback via CMS version history

Details

  • Even after accept in our UI, a human may still publish via the normal CMS process
Two-stage trust: our review UI, then optional CMS editorial publish.
Publish only after approval

Live documentation changes only after explicit human approval — never automatic in MVP.

Responsibilities

  • No auto-publish from merged PRs
  • Audit trail of who approved what
  • Future: confidence-based automation only for low-risk, auditable changes

Details

  • Phase 5 evolution: narrow automation with revert and audit
  • Users should never feel AI secretly changed documentation
Earn the right to automate more after proving trust.

13 · Building the MVP with online tools

Practical implementation choices — not the architecture itself.

Frontend / Review UI

  • Next.js
  • Hosted on Vercel or similar
  • Shows current doc, suggested diff, evidence, confidence, actions

Backend / API

  • Next.js Route Handlers, Node API, or lightweight backend service
  • Handles webhooks and CMS API calls

Database

  • Postgres via Supabase, Neon, RDS or similar
  • Stores events, suggestions, feedback, audit logs

Queue / Jobs

  • Inngest, Trigger.dev, Upstash QStash, BullMQ, SQS or similar
  • Handles async analysis and retries

Search / Retrieval

  • Start with Postgres full-text search + metadata matching
  • Add embeddings later only if needed

AI Provider

  • OpenAI, Anthropic or similar
  • Used for summarisation and suggestion drafting
  • Never owns publishing decision

CMS Integration

  • Read docs via CMS API
  • Write accepted changes as drafts only

Observability

  • Sentry, logs, job dashboard
  • Track failures, latency, rejected suggestions

Auth / Permissions

  • Auth0, Clerk or existing auth
  • Check reviewer permissions before allowing updates
These tools are implementation choices, not the architecture itself. The architecture should remain portable.

Phase 5 · Confidence-based automation
  • Low-risk updates only · still auditable · human can revert

  • Multiple repos · multiple documentation sources · release-based workflows

21 · Product metrics

  • Suggestion acceptance rate
  • Edit-before-accept rate
  • Rejection reason distribution
  • Docs updated per merged PR
  • Review latency
  • Time saved per documentation update
  • Reduction in stale docs
  • User trust score
  • False positive rate
  • False negative review
  • % of suggestions with evidence
  • % of stale docs detected by reconciliation
Do not judge success only by number of AI suggestions generated. The better measure is whether users accept, trust and act on useful suggestions.

22 · Trade-offs

Precision vs recall

Early MVP should favour precision to avoid noisy suggestions.

Drafts vs auto-publish

Drafts build trust; auto-publish can come later for low-risk changes.

Live CMS query vs indexed docs

Index improves speed and search; must handle freshness.

Keyword search vs embeddings

Keyword/metadata simpler; embeddings help semantic matching but add complexity.

Rules vs LLM generation

Rules constrain; LLM drafts natural language; combine both.

MVP vs platform

Prove workflow first before scaling integrations.

Speed vs trust

Faster automation is useless if users do not trust it.

23 · Security & permissions

  • Validate webhook signatures.
  • Store secrets safely.
  • Respect CMS permissions.
  • Do not expose private repo details to unauthorised users.
  • Audit all accepted changes.
  • Avoid sending sensitive code/docs to AI provider unless allowed.
  • Support tenant isolation if multi-tenant.
  • Admins can disable automation.
  • Redact secrets from prompts.
Trust Security is part of user trust.
Start with a system users trust and manually review — then earn the right to automate more later.
