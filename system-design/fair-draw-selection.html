System Design

Fair Draw Winner Selection System

A trustworthy, auditable and scalable system for a prize-draw platform: selecting winners from high-volume prize draws.

Core framing: The hardest part is not generating a random number. The hardest part is proving to customers, auditors, regulators and the business that the winner was selected fairly, securely and reproducibly.
01

Architecture — Simple View

Entries are validated and stored, frozen into an immutable snapshot at draw close, then the winner is selected against that snapshot, logged, reviewed, and notified. This is the version I would draw first on the whiteboard.

Numbered 1–9 in flow order. The accent colour highlights the trust pivot: the snapshot is frozen at draw close, and winner selection runs only against it. Observability spans the whole pipeline.

02

Architecture — Detailed View

The same trust pivot, but with the real-world plumbing: an event-driven ingestion path, idempotent consumers, WORM stores, dual-approved closure, an idempotent selection worker, and a notification retry path. I would only draw this if asked to go deeper.

Same backbone as the simple view, expanded: ingestion is event-driven with idempotent consumers; stores are append-only / WORM; closure is dual-approved; the selection worker is idempotent on draw + snapshot + algorithm version; and notifications fail into a retry queue / DLQ. The accent-coloured path is still the only thing that decides a winner.

03

Scaling Under Pressure

The interviewer will keep raising the numbers. The goal is to show you can spot the bottleneck and evolve the storage model — without ever losing auditability or reproducibility.

“30M entries”Store every ticket

One row per entry. Simple, trivially auditable, and 30M rows is comfortable for a normal database.

  • Holds up: writes, snapshot, indexed lookup all fine.
  • Selection: pick number n, look up that row by primary key.

“500M entries”Store entry ranges

Now per-ticket rows hurt: storage, snapshot size and write throughput all balloon. Switch to one row per purchase holding a contiguous range.

  • Bottleneck: row count and snapshot size, not the random draw.
  • Selection: binary search the sorted ranges for n — O(log N), no full scan.

“one buys 100k”Cumulative counts

A single customer buying 100,000 entries is just one range row — ranges already absorb skew. Cumulative counts per owner make weighted selection and verification cheap.

  • Bottleneck: hot keys / skew, handled by ranges + idempotent writes.
  • Selection: map n through cumulative totals to an owner.

How I would answer the escalation: start with individual entries because it is the easiest to reason about and audit. As volume grows I move to entry ranges (one row per purchase) so storage and snapshot size scale with purchases, not tickets — and a 100k-entry buyer becomes a single row rather than a hot spot. Cumulative counts are the next step for weighted or extreme-scale cases. Throughout, the winning number maps deterministically through the snapshot with a binary search rather than a full scan, and the snapshot + seed + algorithm version still make the result fully reproducible.
▸

Name the bottleneck explicitly — at scale it is row count, storage and snapshot size, not generating the random number.

▸

Never scan hundreds of millions of rows to pick a winner; map the number through sorted ranges or cumulative totals in O(log N).

▸

Whatever model you pick, auditability and reproducibility are the binding constraint — efficiency must not break them.

▸

Adapt incrementally: individual → ranges → cumulative, and only move when the numbers force it.
