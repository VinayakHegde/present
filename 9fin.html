Tech Q&A

Walk me through data flow end-to-end
Warehouse facts + Postgres definitions → calc service applies rules → validation → FastAPI → BFF shapes → React/export. Browser never hits calc service. Invalidation on definition or source refresh.
Why server-side calculation?
Single source of truth. Client-side duplicates logic and drifts — trust incident if UI and export disagree.
How do you guarantee correctness?
Golden fixtures from customer models; validation at calculation boundary; CI on calc paths; visible failure not silent wrong numbers.
Caching vs live calculation?
Two tiers — not either/or. Pinned metrics: async snapshots (~120ms). Ad-hoc: live calc + React Query dedupe. lastCalculatedAt makes staleness explicit.
Scale 10×?
Hot path scales with snapshot reads. Cold path needs warehouse indexing + queue-backed recompute. Architecture already separates tiers.
Idempotency?
Materialisation keyed by (tenantId, definitionId, scopeHash, sourceWatermark). Upsert idempotent; dedupe lock on async worker.
Mentoring example?
4-layer test template with junior FE; they shipped IRR + export tests independently; checklist in PR template.
Biggest near-miss?
Nearly shipped client-side calc — killed after spike showed Excel-parity failure on FX month-end edge case.
How do you use AI?
Accelerator for scaffolding, tests, mocks — not architecture or financial rules. Full CI; no customer data in prompts; I own every merge.
What would you do differently?
Instrument adoption/export analytics in PRD day one (backfilled post-GA). Run data-platform warehouse spike in discovery, not mid-build.
