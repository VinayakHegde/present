★How to use this in the interview

This is not a product demo script. It is a short guide for discussing architecture, productionisation, trade-offs, and how AI was used.

Suggested flow

  1. Frame the problem — affordability reflection for vulnerable customers; not arrangements or agent tooling.
  2. Architecture — layered design, route groups, what would change for production.
  3. Productionisation — auth, Postgres, observability, share-link hardening.
  4. Trade-offs — what I would defend vs change; biggest risks.
  5. AI workflow — structured phases, review, what I pushed back on.

Three sentences to anchor the discussion

  • Architecture: “Pure domain logic, a single persistence boundary, thin server pages, sync UI components — explainable now, swappable later.”
  • Production: “The take-home proves the journey; production adds real identity, durable storage, and controls we deliberately deferred.”
  • AI: “AI sped up drafting; I still owned scope, ran review passes, and kept prompt history so you can see what I accepted vs rejected.”

1.Overview

Customer Financial Health is a customer-facing affordability reflection and tracking app. A customer sees their disposable income, an affordability band, plain-language explanation, change vs the last submission, and a support signpost. Each submission creates an immutable snapshot; history is browsable. Stretch goals shipped: currency/country, time-limited share links, and PDF export.

The brief asked for assessment plus tracking — not repayment-plan booking, collections workflow, or agent tooling. I dropped an early AI suggestion to build an arrangements flow and kept scope tight: one calculation model, append-only snapshots, and a simple persona selector for the demo.

2.Key decisions

Product

  • Reflection, not advice — customers see their own figures explained back; no plan booking or regulated-advice framing.
  • Supportive vulnerable-customer posture — calm tone, visible support signpost, no automated vulnerability classification from numbers.
  • Immutable history — corrections are new submissions; prior snapshots stay visible.

Technical

  • Next.js App Router — server-rendered customer journey; route groups separate the main app from the share recipient view.
  • SQLite for the demo — file-based, survives restarts; Postgres is the first production swap behind the same repository boundary.
  • Server Actions for forms — persona pick, I&E update, share-link mint; PDF is the only binary route handler.
  • Layered code — pure affordability logic, one DB module, thin async pages, sync presentation components.

3.Architecture

The app is organised in four layers:

  • Domain — affordability calculation, validation, and customer-facing copy. No database or React dependencies.
  • Persistence — SQLite via Drizzle; snapshots and share links. The only place that touches the database.
  • Server I/O — pages read the persona cookie and repository; Server Actions handle mutations.
  • UI — sync components render dashboard, history, forms, and shared statement views.

Route groups keep concerns separate: (main) is the customer journey with navigation; (share) is the recipient view with no persona chrome; (case-study) is this internal notes page.

Production evolution is mostly boundary swaps: signed sessions instead of a demo persona cookie, Postgres instead of SQLite, edge hardening on share URLs — not a rewrite of the domain or page structure.

4.Productionisation

What would need to ship before real customers:

  • Real authentication — replace the unsigned persona cookie; re-validate on every mutation.
  • Postgres + backups — HA storage, connection pooling, point-in-time recovery.
  • Retention and erasure — TTL on snapshots and share links; right-to-erasure workflow.
  • Share-link hardening — single-use or revocation, rate limits, consent step before mint.
  • Observability with redaction — metrics on usage patterns; never log income/expenditure values or bearer tokens.
  • Access-log hygiene — strip or hash share and PDF URLs at the reverse proxy.

What stays the same: pure calculation logic, append-only snapshots, reflection-not-advice framing, and the layered layout.

5.Trade-offs and gaps

Happy to defend for a take-home

  • SQLite — zero ops for a reviewer, real migrations, data persists across restarts.
  • Persona cookie — exercises the journey without standing up an IdP.
  • Layered architecture — production changes are boundary swaps, not rewrites.
  • Structured AI workflow — caught scope creep and doc/code mismatches before they landed in code.

Would change first in production

  • Auth and Postgres before any new feature surface.
  • Share tokens out of URL paths where possible, to reduce access-log leakage.
  • Browser-level smoke tests for the full submit-and-redirect journey.

Biggest risks today

  1. Bearer tokens and snapshot ids in reverse-proxy access logs.
  2. Unsigned persona cookie — not a real threat model.
  3. No retention job — storage limitation not operationalised.

6.How I used AI

Approach

I worked in phases — understand the problem, write requirements and design, then build in small vertical slices. Cursor with version-controlled rules, a read-only critic pass between phases, and full chat transcripts in the repo.

What review caught

  • Dropping an arrangements workflow the brief did not ask for.
  • Adding an explicit “not financial advice” notice before it was buried in the UI.
  • Removing a free-text note field (PII risk).
  • Designing share-link security before implementation.

Where I pushed back on AI

  • Rejected agent tooling, real auth, and Playwright for the MVP time box.
  • Did not trust library marketing claims without checking (e.g. PDF renderer weight).
  • Did not invent regulatory citations — paraphrases are labelled unverified in discovery notes.

What I would tighten next time

Automated checks that docs and code stay aligned; mandatory critic on design changes; copy linting so supportive tone does not drift. Full prompt history is in the repo if you want to audit decisions.
