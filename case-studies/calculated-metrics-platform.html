Approach
The latency decision was a two-tier read model. Pinned dashboard metrics take the hot path: when a definition is saved we async materialise a validated snapshot to Postgres - dashboard reads are a simple lookup, about 120ms p95. We show lastCalculatedAt so users know the freshness contract. Ad-hoc or first-time views take the cold path: live warehouse calculation, validation at the boundary, React Query dedupes identical requests and keepPreviousData avoids blank flashes on refresh. If the user pins that metric, the next save promotes it to the hot path. Invalidation is explicit: definition change or upstream data refresh triggers re-materialisation - we accepted seconds of staleness on pinned metrics to buy sub-second dashboard UX. Same trade-off you'd see on syndication: pre-compute stable feeds, live path for exceptions.

Technologies & why

Patterns first · production stack · contract-first APIs · 4-layer test parity

Approach
Lead with patterns, not languages. Hiring Manager Deep-Dive Q&A

For rehearsal only. Answer conversationally - do not sound rehearsed.

1. Walk me through data flow from source facts to displayed metric.
Warehouse facts + definition config → calculation service applies rules → validation → BFF shapes → UI/export. Browser never hits calculation service directly.
2. Why server-side calculation?
Single source of truth; client/export-then-calculate duplicates logic and drifts. Trust incident if two surfaces disagree.
3. How did you model metric definitions?
Versioned JSON schema in Postgres: metricType, scope (fund / deal / portfolio), aggregation, filters (date window, currency), formula refs, validation rules. PM fills name + description; engineer reviews before production enable.
4. How is the configuration driven for calculated metrics?
The metric definition is stored as configuration, not hard-coded per metric. A definition includes fields such as metric type, scope, filters, aggregation method, date window, currency handling, formula references, and validation rules. The calculation engine reads that definition and applies it server-side, so adding a new metric usually means adding or extending a definition and tests rather than creating a one-off implementation path.
5. Who provides the calculations or formula, and how does that affect load?
The formula comes from the business or domain side, usually Product, Finance/Ops, Customer Success, or an existing customer model. My role was to turn that into an engineering-safe definition: validate assumptions, document edge cases, create golden fixtures from known examples, and make sure the output matched expected results before exposing it to users. From a load perspective, not every metric is calculated live every time. Frequently used or pinned metrics are materialised asynchronously and served from snapshots. Ad-hoc metrics are calculated on demand, validated, and deduped or cached at the request layer.
6. How does this architecture scale as more metrics are added?
Scalability comes from separating metric definitions, calculation execution, validation, and serving. New metrics follow the same definition and test template rather than creating isolated code paths. High-usage metrics can be promoted to the materialised path, while exploratory metrics stay on the live calculation path. If volume grows further, the next step would be queue-backed recalculation, partitioned snapshots, and prioritised recompute jobs.
7. How do you guarantee correctness when rules change?
Golden fixtures from known models; validation layer; CI on calculation paths; visible failure not silent wrong numbers.
8. Wrong or incomplete source data?
Reject at calculation boundary: null valuation → ValidationError with field path; missing FX rate → block export, surface "missing rate for {pair} on {date}"; partial portfolio → exclude rows + audit log - never silent blend or approximate.
9. Scale 10× portfolios / metrics?
Hot path scales with snapshot reads (O(1) lookup). Cold path needs warehouse indexing + async materialise for anything pinned. At 10× I'd add snapshot partitioning and queue-backed recompute - architecture already separates the two tiers.
10. Idempotency / replay?
Materialisation jobs keyed by (tenantId, definitionId, scopeHash, sourceWatermark). Upsert is idempotent; safe replay on same watermark; async worker with dedupe lock - re-run after definition change doesn't double-write snapshots.
11. Map to our Node/GCP serverless stack?
Calc engine → Node Cloud Functions / Cloud Run (stateless workers). Definitions + snapshots → Postgres. Warehouse reads → BigQuery mart (read-only). BFF → Next.js Route Handlers. Invalidation → Pub/Sub on definition save / mart refresh. Same validation-at-boundary + Sentry/Grafana per feed - identical muscle to syndication transforms.
12. Biggest architectural near-miss?
Nearly shipped client-side calculation for faster dashboards - killed after a 2-hour spike showed Excel-parity failure on an FX month-end edge case. Would have been a trust incident; reinforced server-side-only rule.
13. Caching vs live calculation?
Not either/or - two tiers. Pinned dashboard metrics: async materialised snapshots (~120ms p95). Ad-hoc: live calc with React Query dedupe. Invalidation on definition or source refresh. lastCalculatedAt in UI makes staleness explicit - users trust the number because they see when it was computed.
14. Testing rule combinations?
Golden fixtures from customer Excel models - one per edge case (null valuation, FX pair, window boundary, roll-up level). Property-style tests on definition combinations so a new metric type can't ship without covering the same four failure modes. Layered CI on every change.
15. Observability in prod?
Sentry on calc failures and FE errors; Grafana dashboards on latency and error rates; funnel on metric usage.
16. Rollback / feature flags?
Tenant-level flag calculated-metrics for GA; per-metric-type flags on backlog. Rollback = disable flag - UI falls back to legacy export path; snapshots retained for forensics. Design partners mid-sprint before all-tenant rollout at sprint end.
17. Versioning for export consumers?
API versioning + export format stability; breaking changes behind new version.
18. Tenancy / who sees which metrics?
Auth service + per-user visibility; BFF enforces scope.
19. Prioritisation with PM when scope was huge?
Phased MVP - investor-ready subset first; explicit definition-of-done.
20. Stakeholder disagreement?
PM wanted 10 metric types in sprint one; finance needed every number traceable to a model. I facilitated and wrote the scope doc: MVP = 3 KPIs (IRR, MOIC, net return) with golden fixtures + validation metadata as DoD; breadth + configurable defs to iteration 2. Finance signed off because "investor-ready" was testable, not subjective.
21. How measured success?
Product analytics: export adoption (~65% fund-ops at 5 weeks). CS ops: manual-reporting tickets ~30% down month one. Time-study: quarterly pack 4–6h → ~20min. Qualitative: customers quoting in-product IRR in board materials.
22. What would you do differently?
Two things: instrument export/adoption analytics in the PRD on day one - we backfilled post-GA and relied on CS for some impact numbers. And run a data-platform warehouse spike in discovery - the valuation mart dependency surfaced mid-build; the hand-off worked, but earlier would have been calmer.
23. Why did you choose this project?
It wasn't a UI tweak or a single API - it required understanding a business problem, defining rules, designing architecture, validating correctness, coordinating stakeholders, and owning delivery through production. It demonstrates senior ownership: product thinking, architecture, and delivery in one.
24. Built It, Own It example?
I merged, deployed to K8s, and was on-call through the design-partner phase. Pre-GA FX boundary alert in Sentry - I triaged (missing month-end rate), added golden fixture, hotfixed before sprint-end all-tenant rollout. Watched Grafana latency + error dashboards; zero P1 at GA.
25. Mentoring example?
Three working sessions with junior FE on 4-layer test template: I wrote first MOIC calculation hook test; they shipped IRR + export component tests independently. Published metric-definition checklist (schema → fixtures → validation → export) in squad PR template - next metric type didn't need bespoke setup.
26. Unblock another team?
Data platform squad owned valuation mart. I wrote 3-option ADR: (A) read mart read-only - recommended for MVP; (B) new materialised view - their backlog for scale; (C) duplicate facts in domain service - rejected (drift risk). One review meeting; neither team blocked.
27. What did you deliberately not build?
Client-side calc (correctness - killed after spike). Generic chart builder (design-system viz sufficient). Self-serve metric authoring UI (explicitly iteration 2, not sprint-one scope creep).
28. New marketplace syndication tomorrow - what transfers?
Validation at boundary, contract-first APIs, phased rollout, observability, optioned cross-team hand-offs.
29. How do you use AI day-to-day?
Scaffolding, tests, mocks, exploration - not architecture or rules sign-off.
30. Guardrails?
CI gate, no secrets in prompts, I own merge and prod behaviour.
31. AI for syndication work?
Yes for fixtures and mapping boilerplate; no for compliance rules and prod config without review.

Focus Themes - Demonstrating Depth + Product Mindset

Six lenses the panel scores on. Every technical choice should trace back to customer trust / correctness - that is the product currency here.

1 · Requirement clarification

What I clarified before writing code
Which KPIs count as "investor-ready"? What is the source of truth for each formula, and what does "correct" mean (parity with which model)? Who consumes the output - dashboard, export, or board pack - and what's the freshness expectation? Turned a vague "add calculated metrics" ask into a versioned definition spec + acceptance criteria, and surfaced the hidden valuation-mart dependency early through discovery questions.
How I made requirements testable
DoD wasn't "looks right" - it was 3 KPIs (IRR, MOIC, net return) with golden fixtures + validation metadata. "Investor-ready" became objective, so finance could sign off and engineers knew exactly when they were done.

2 · Technical trade-offs

Server-side vs client-side calculation
Chose server-side for a single source of truth over the faster client-side dashboard. Client calc duplicates logic and drifts → two surfaces disagree → trust incident. Validated by a spike that found an FX month-end parity bug.
Two-tier read model (hot vs cold)
A freshness / cost / latency trade-off, not a tech flex. Pinned metrics = materialised snapshots (~120ms p95); ad-hoc = live calc. lastCalculatedAt makes staleness explicit so users trust the number.
Read-only mart vs duplicating facts; boring stack vs Kafka/Spark
Read the valuation mart read-only to avoid data drift, accepting some coupling. Picked Postgres + managed queue + workers over Kafka/Spark - operability now, with clear seams (queue-backed recompute, snapshot partitioning) to split when a real bottleneck appears.

3 · Security

Multi-tenant isolation
Every query scoped by tenantId with row-level security; BFF enforces per-user visibility; the calculation service is never exposed to the browser. Shared DB with RLS to start, isolate noisy tenants only when needed.
Untrusted ingestion (PDF / Excel / mailbox / Slack)
External documents are untrusted input: sandboxed extraction, file type/size limits, malware scan, sanitise before parse, never execute. Bad/partial data is rejected at the calculation boundary with a field-level ValidationError - never a silent blend.
Data protection, secrets, auditability
Financial/PII data → encrypted at rest (object store + DB), TLS in transit, least-privilege service accounts. Exports via signed URLs with expiry. No secrets in code or AI prompts - secret manager only. Audit log on metric reads/exports; the trust boundary means a human verifies before a metric is treated as trusted.

4 · Testing

Golden fixtures + 4-layer template
Golden fixtures from real customer Excel models - one per edge case (null valuation, FX pair, window boundary, roll-up level). A 4-layer template (calc hook → validation → API contract → export) means a new metric type can't ship without covering the same four failure modes.
Combinations, idempotency, fail-visibly
Property-style tests across definition combinations; replay/idempotency tests on materialisation jobs keyed by (tenantId, definitionId, scopeHash, watermark). CI gates the calc paths; validation fails visibly rather than serving a silently-wrong number.

5 · AI usage

Where AI helped - and where it didn't
Used: scaffolding, test + fixture generation, mocks, exploration - PRD/SDD locked first. Not used for sign-off: calculation rules, security boundaries, data-model and contract decisions. AI accelerated the work; I owned the judgement.
Guardrails & traceability
No secrets in prompts; full CI gate on AI-assisted code; engineer owns merge and production behaviour. Kept prompt history / session snapshots so AI usage is auditable, not a black box.

6 · Product / customer impact

The problem in customer terms
Customers didn't trust in-product numbers, so they rebuilt derived KPIs in spreadsheets - slow, error-prone, and eroding confidence in the platform. Trust, not features, was the real gap.
Measured outcomes
~65% adoption in 5 weeks; ~30% fewer manual-reporting tickets; quarterly pack 4–6h → ~20min; customers quoting in-product IRR in board materials. Every architectural decision (server-side, validation, two-tier, trust boundary) traces back to protecting the customer from acting on a wrong number.
