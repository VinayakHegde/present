Two-tier read model technical trade-off

S: Live calculation of every metric was too slow for dashboards (3–8s); full pre-computation was too stale.
T: Fast reads without lying about freshness.
A: Two tiers — pinned/frequent metrics served from materialised snapshots (~120ms p95); ad-hoc metrics calculated live. UI shows lastCalculatedAt so staleness is explicit, with client-side cache dedupe on hot paths.
R: Sub-second dashboards with an honest freshness contract. Users trust what they see; never show stale data as live.

Use when asked about: a trade-off you made · real-time / perceived performance · caching decisions · designing for data-heavy UI

Pre-GA production incident production ownership

S: On-call through the design-partner phase before general rollout.
T: Catch correctness issues before they reached all customers.
A: A monitoring alert fired on an FX boundary case. A missing month-end rate caused a visible validation failure rather than a silent wrong number. Triaged, added a golden fixture for the pair/date, hotfixed before the all-tenant rollout; watched dashboards through launch.
R: Zero P1 at GA. The failure mode worked as designed: visible failure, not silent drift.

Use when asked about: a production incident · on-call and reliability · quality under pressure · observability in practice

Legacy migration via feature flags safe rollout

S: The new metrics had to ship without breaking existing export/reporting workflows customers depended on.
T: Migrate customers from the legacy manual-export path safely.
A: Tenant-level feature flag. UI falls back to the legacy export on rollback. Old snapshots retained for forensics. Two design-partner accounts before all-tenant GA.
R: Zero P1 at launch; a real rollback story; design partners caught the FX edge case pre-GA.

Frame as strangler / incremental migration — not big-bang rewrite.

Use when asked about: legacy migration · de-risking a launch · incremental delivery · learning from customers

Technical leadership — specs, ADRs, contracts tech lead

S: Platform feature crossing team boundaries, with an undecided hand-off to the data platform.
T: Set technical direction the squad could execute and reuse.
A: Authored the SDD. Drove contract-first OpenAPI with generated clients. Wrote a 3-option ADR for the data-platform hand-off — chose read-only mart for MVP rather than blocking on the ideal integration. Established the 4-layer test checklist in the squad PR template.
R: Next metric type needed no bespoke setup; patterns reused on later features; deployed and on-call through launch.

Use when asked about: technical leadership · influencing architecture · driving standards · making progress amid unknowns

Short supporting answers (local only)

How do you operate in ambiguity?

Convert ambiguity into a written, testable scope doc. Facilitate alignment between breadth (PM) and auditability (stakeholders). Time-box spikes for unknowns — the two-hour client-side-calc spike; a half-day data-platform spike I'd run earlier next time. Ship to design partners, not all tenants, when correctness risk is high.

"I don't wait for perfect requirements — I write the smallest testable scope doc, get sign-off, and ship to learn."

How do you learn from customers?

Design partners before full rollout — they surfaced FX and roll-up edge cases. Golden fixtures built from real customer models, not synthetic data. Post-launch: export-event analytics, support-ticket tags, time studies on reporting workflows. Qualitative signal: customers quoting in-product numbers in their own board materials.

What did you deliberately not build?

Client-side calculation (correctness). A generic chart builder (existing viz sufficient). Self-serve metric authoring UI (iteration two, not sprint-one scope creep). Saying no protected the definition of done and the launch date.

What would you do differently?

Instrument adoption/export analytics in the PRD on day one — we backfilled some analytics post-GA. Run the data-platform spike in discovery, not mid-build.
