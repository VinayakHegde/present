When designing a new service, I would start with the problem and the users rather than the technology.

First I would clarify what the service is meant to achieve, who depends on it, and what success looks like. Then I would look at the core data model, API boundaries, expected traffic, security requirements, failure modes, and how the service will be monitored in production.

After that I would think about trade-offs: does this need to be synchronous or asynchronous, what needs to be strongly consistent, what can be eventually consistent, and what is the simplest version that solves the problem without creating unnecessary complexity.

For a production service I would also think early about testing, observability, deployment, rollback, and ownership. I want the team to understand not just how to build it, but how it behaves when things go wrong.

Senior add-on: I try not to optimise for scale too early, but I do want to make sure the design does not block obvious future growth.
What they are testing System design maturity, product thinking, non-functional requirements front of mind.

I think sustained code quality comes from a combination of shared expectations, tooling, code review, and team culture — not from any single gate or checklist.

At a practical level, quality means code that is readable, testable, and safe to change. I like clear conventions around TypeScript, testing, accessibility, API contracts, error handling, and component structure. Tooling helps with the basics: linting, formatting, type checking, tests in CI, and review gates. But that is a baseline, not the full definition of quality.

Code review is where the team applies judgement. I try to make reviews constructive and focused on long-term maintainability, not personal preference. If something is hard to test, risky from a security perspective, likely to confuse the next engineer, or solving a one-off problem in a non-reusable way, I will call that out and explain why.

I also think quality has to be taught, not just enforced. Documenting patterns, sharing examples, and helping engineers understand the reasoning behind decisions is how teams keep standards high as the codebase grows.

Unit tests check small pieces of logic in isolation. Beyond that, I would think about several layers.

  • Integration tests — multiple parts working together, e.g. a component talking to an API client or a backend service interacting with a database
  • Contract tests — useful when frontend and backend teams work against APIs; ensure the API shape matches what consumers expect
  • Component tests — UI behaviour more realistically than pure unit tests: forms, loading states, error states
  • End-to-end tests — critical user journeys through the system; use selectively — slower and more brittle
  • Also consider: accessibility tests, visual regression, performance tests, smoke tests after deployment, security testing depending on risk profile
Senior add-on: I try to avoid relying on only one test type. The right approach is usually a layered strategy: fast unit tests for logic, integration tests for boundaries, and a small number of E2E tests for business-critical flows.

The key is to build quality into the workflow rather than treating it as a final gate.

I like small PRs, clear acceptance criteria, good automated checks, and early conversations about edge cases. That avoids discovering quality issues at the end.

Reusable patterns help velocity. If the team has shared approaches for API calls, error handling, testing, accessibility, and feature flags, engineers move faster because they are not reinventing decisions every time.

AI can also help here, especially for generating tests, mocks, boilerplate and documentation. But I would still keep human review on architecture, security, accessibility and production behaviour.

The balance is not speed versus quality. Good engineering practices usually improve both over time.

I use AI as an accelerator, not as a replacement for engineering judgement.

Day to day, I use tools like Cursor and Claude Code for scaffolding, test generation, refactoring ideas, documentation, exploring unfamiliar code, and speeding up repetitive implementation work.

Where I am careful is architecture, business logic, security boundaries, data handling, and production decisions. I always review AI output as I would review a junior engineer's code: useful, but not automatically trusted.

The biggest benefit I see is that AI lets engineers spend less time on repetitive work and more time on understanding the problem, making better design decisions, testing properly, and communicating trade-offs.

"AI helps me move faster, but I still own the outcome."

Best story: Calculated Metrics

One of the most interesting problems I worked on was a calculated metrics platform.

Business problem: Customers were relying on manual spreadsheets to calculate important derived metrics. Different teams could end up with different numbers, which created trust issues and support overhead.

Technical challenge: The metrics needed to be configurable, correct, performant, and available in the product. We had to decide where calculations should happen, how to validate correctness, and how to keep the user experience fast.

What I led: Server-side calculation to keep one source of truth; validation boundaries and golden fixtures based on known customer examples; two-tier read model — live calculation for exploration and materialised snapshots for frequently used metrics.

Result: More trusted in-product workflow, reduced manual reconciliation, better performance for common views, and fewer support requests.

Emphasise: correctness over convenience · validation before serve · performance trade-offs · measurable product outcome

Best story: AIAD / testing workflow

One DX issue I noticed was that engineers were repeatedly spending time creating similar test scaffolding, mocks, fixtures and API integration boilerplate.

The problem was not just time lost. It also led to inconsistent test coverage and different teams solving the same problem slightly differently.

I helped introduce a more structured specification-driven workflow. We would define the behaviour and contracts first, then use AI-assisted tooling to accelerate repetitive implementation such as mocks, test scaffolding and boilerplate.

I also encouraged reusable templates and review checklists so engineers knew what "good" looked like.

Result: Faster delivery, more consistent implementation, and better confidence that changes were tested across the right layers.

"I see DX as a force multiplier: if you improve the workflow, every future feature becomes easier to ship safely."

Best story: Calculated Metrics MVP scoping

On the calculated metrics project, the initial scope was broad. There were many possible metric types and reporting capabilities stakeholders wanted.

The risk was that if we tried to build everything, we would either miss the delivery window or compromise correctness, which would damage trust.

I worked with Product and business stakeholders to narrow the first release to the highest-value metrics and the core export workflow. We agreed that correctness, validation and customer trust were more important than breadth.

This meant some features, such as more advanced self-serve metric authoring or richer visualisation, moved to later iterations.

Result: We shipped a smaller but more reliable version, validated it with users, and created a foundation we could extend.

Senior angle: Cutting scope was not about doing less. It was about protecting the outcome.

I prefer mentoring through practical collaboration rather than only giving advice.

For example, I've paired with less experienced engineers on testing patterns, architecture decisions and code reviews. Initially I might work closely with them on the first implementation, explaining why we structure the code or tests in a certain way. Then I gradually step back and let them apply the pattern independently.

I try to make feedback specific and constructive. Instead of saying "this is wrong", I explain the trade-off: maintainability, accessibility, security, performance or readability.

The goal is not to make engineers dependent on me. The goal is to help them build judgement so they can make good decisions independently.

People should be allowed to make mistakes in environments where the impact is controlled and the learning is real.

For example, mistakes in local development, test environments, small reversible changes or internal tooling can be valuable learning opportunities. But mistakes involving customer data, payments, security or production reliability need stronger guardrails.

So I do not think the answer is "mistakes are always fine" or "mistakes are never acceptable". The job of a good engineering culture is to create safe boundaries: code review, testing, feature flags, monitoring, rollback plans and blameless learning.

If something goes wrong, I care about whether we understand why it happened and whether we improved the system afterwards.

"The goal is not zero mistakes. The goal is controlled risk and fast learning."

Best story: Client-side vs server-side calculation

One disagreement I had was around where calculation logic should live for a product feature.

There was an argument for doing more on the client side because it would be quicker and simpler for the UI. But I was concerned that this could create inconsistent results between the UI, exports and backend systems.

Rather than making it about personal preference, I reframed the discussion around the customer outcome: users needed to trust the numbers.

We compared the options against correctness, performance, maintainability and user trust. Based on that, we chose server-side calculation with validation, even though it required more initial work.

Outcome: A more reliable system, and the team had a clearer shared understanding of why the decision was made.

Avoid: "I convinced them I was right."
Say: "I aligned the team around the trade-offs and outcome."

I admire engineers who combine curiosity with ownership.

The best peers I've worked with are not just technically strong; they ask good questions, care about the customer outcome, and take responsibility beyond their own ticket.

I also admire people who can challenge ideas constructively. Strong teams need people who are willing to disagree, but in a way that improves the decision rather than creating friction.

One thing I often encourage peers to improve is surfacing assumptions and trade-offs earlier.

Sometimes engineers jump into implementation before the team has fully agreed what problem is being solved, what success looks like, or where the risks are.

I try to encourage more upfront clarity through lightweight design notes, ADRs or technical discussions. Not heavy process, but enough alignment to avoid rework later.

"I would like teams to make implicit assumptions explicit earlier."

Frame as constructive, not critical.

React / frontend — look for

  • Missing error handling, loading and empty states
  • Incorrect key usage in lists
  • Unsafe any — prefer proper types
  • State update issues — functional updates where safer
  • Too much state that React Query could own
  • Component too large or mixing concerns
  • Reusable vs one-off logic
  • Accessibility: button vs link, divs used incorrectly, missing labels, focus states, alt text

Testing — mention

  • Unit tests for pure logic
  • Component tests for UI behaviour
  • Integration tests for API boundaries
  • E2E tests only for critical flows
  • Accessibility tests where relevant
  • Contract tests if API is shared

Performance — mention

  • Pagination / lazy loading
  • Avoid rendering huge lists
  • Memoisation only where useful
  • React Query caching
  • Backend-side filtering/pagination if dataset grows

Security — mention

  • Permissions before delete / bulk delete
  • Destructive actions need confirmation
  • Server-side validation — do not trust client-only checks
  • Audit logs for sensitive actions
