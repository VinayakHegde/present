Near Real-Time Stock Price Alert System

You receive a continuous time series of stock prices. When a price crosses the threshold a user set in an alert, notify them — within a fraction of a second, at the scale of millions of alerts across thousands of symbols.

Core framing: The hard part is not comparing two numbers. It is (1) detecting a crossing rather than a level — which inherently needs the previous price as state — and (2) matching a single tick against millions of user alerts in near real time, while delivering each notification exactly once and surviving feed gaps, out-of-order ticks and restarts.

01

Requirements & Framing

Pin down what "alert when the price crosses" actually means, and the scale we are designing for, before drawing anything.

Functional

  • User creates an alert: (symbol, target, direction) — e.g. AAPL crosses above 200.
  • We consume a stream of (symbol, price, ts) ticks.
  • Fire when the price crosses the target, not merely when it sits past it.
  • Deliver a notification (push / email / in-app) once per crossing.
  • Alerts are one-shot by default (disarm after firing); optionally repeating with re-arm.

Non-functional

  • Near real-time: tick → notification in well under a second.
  • Scale: millions of active alerts, thousands of symbols, high tick rates on hot names.
  • Correctness: never miss a crossing, never fire a phantom one, never double-send.
  • Resilient: tolerate feed gaps, out-of-order / duplicate ticks, and worker restarts.
  • Durable: alerts survive crashes; in-memory state is rebuildable.

02

The Crossing Problem — the real insight

A crossing is defined by a transition, so detecting one is inherently stateful: we must compare the new price against the last price we saw for that symbol.

A "cross up" alert fires only on the transition from below to at/above the target — not on every tick the price stays above. Detecting it needs both the previous and the current sample.

Definition

Given previous price p0, new price p1 and target T:

  • Cross up: p0 < T ≤ p1
  • Cross down: p0 > T ≥ p1

The target must lie between the two samples. A single price in isolation can never define a crossing.

▸

The whole design hangs on one fact: we must hold the last price per symbol. That single piece of state turns "compare two numbers" into a real streaming system.

Gaps & jumps

A tick can leap from 195 to 210, skipping 200. The crossing still counts — the interval test (p0, p1] catches it even with no sample exactly at the target.

First tick / no prior

With no p0 we cannot know direction, so we seed last-price and do not fire. Otherwise a restart would re-fire every alert already past its target.

Equality & flapping

Define the boundary explicitly (≤ vs <). For repeating alerts add hysteresis so a price hovering on the line doesn't fire on every micro-tick.

03

Architecture

Ingest and normalize the feed, route ticks by symbol so each symbol's state lives in one place, match against the in-memory alert index, then hand crossings to a durable, deduped notification path. This is the version I'd draw first.

The teal path is the hot loop: feed → normalize → route by symbol → match. Routing by symbol guarantees a symbol's last price and alerts live on one worker, so crossing detection is a local, lock-free operation. The blue path turns crossings into deduped, retriable notifications.

whyPartition by symbol

All state needed to detect a crossing — last price + every alert on that symbol — is co-located. No cross-worker coordination, ordered processing per symbol, trivial horizontal scaling.

whySplit hot path from delivery

Matching must stay fast and CPU-bound; delivery is slow and I/O-bound (push gateways, SMTP, retries). A queue between them absorbs spikes and isolates failures.

04

Matching Engine & Algorithm

Per symbol we keep the last price plus two sorted indexes of thresholds. A tick becomes an interval query: find every alert whose target lies between the old and new price.

Per-symbol state

  • lastPrice — the previous tick for this symbol.
  • upAlerts — "cross above" targets in a sorted structure (balanced BST / skip list).
  • downAlerts — "cross below" targets, sorted.

On a tick we don't scan all alerts. We do a range query over the relevant sorted set — only the alerts actually crossed are touched.

Cost: O(log n + k) per tick — n alerts on the symbol, k alerts actually crossed (usually 0). The random number / comparison is never the bottleneck; the index is. 
function onTick(symbol, price, ts):
  state = stateFor(symbol)
  p0 = state.lastPrice
  if p0 is null:           # first tick: seed only
    state.lastPrice = price
    return

  if price > p0:               # moved up
    fired = state.upAlerts
              .rangeOpenClosed(p0, price)   # (p0, price]
  elif price < p0:             # moved down
    fired = state.downAlerts
              .rangeOpenClosed(price, p0)   # [price, p0)
  else:
    fired = []                # no movement, no cross

  for alert in fired:
    emit(CrossingEvent(
      alert.id, symbol, alert.target, price, ts,
      key = alert.id + ":" + alert.epoch))  # idempotent
    state.disarmOrRearm(alert)

  state.lastPrice = price
▸

Sorted thresholds turn "check N predicates" into "query an index". The same trick a database uses for a range scan, applied to standing queries.

▸

The matching engine is single-writer per symbol, so the index needs no locks. State updates and the lastPrice write happen in one ordered pass.

▸

Creating / editing an alert is just an insert / delete into the sorted set on the owning worker — routed the same way ticks are.

05

Scaling Under Pressure

The interviewer will push the numbers up: more alerts, more symbols, bursty hot names. The answer is to scale the partitioning and the delivery fan-out — never the per-tick comparison.

"more symbols"Shard by symbol

Consistent-hash symbols across matching workers. Each owns a disjoint set; adding workers rebalances partitions. State is rebuildable, so moving a partition just means replaying alerts + reseeding last price.

"a hot symbol"Conflate ticks

One name can dominate the feed. The crossing only depends on first vs last price across a window, so conflate: process the latest tick and skip the intermediate ones — without missing the crossing, since the interval still spans them.

"millions of alerts"In-memory index

The sorted index per symbol lives in RAM for speed, backed by the durable Alert Store. Even 10M alerts is a few GB spread across shards. On restart, rebuild from the store.

How I'd answer the escalation: the per-tick work is already O(log n + k), so I never scale that — I scale partitions (more matching workers, symbols hashed across them) and delivery (a queue plus horizontal notification workers). Hot symbols are handled by conflation on the read side and price-band sub-partitioning on the alert side. State is always rebuildable from the durable Alert Store, so any worker can fail and recover without missing or duplicating crossings.
06

Correctness & Delivery

In a financial product the failure modes are the interview. Each crossing must fire once, never duplicate, and survive messy real-world data.

Exactly-once feel

True exactly-once is impossible end-to-end, so: detect once (single-writer per symbol), attach an idempotency key (alertId : armEpoch), and dedupe at the notification service. Delivery is at-least-once + dedupe → user sees one alert.

One-shot vs repeating

Default: disarm on fire so it can't re-trigger. Repeating alerts re-arm only after the price moves back across by a hysteresis band, preventing flap-firing around the line.

Out-of-order & duplicate ticks

The feed isn't perfectly ordered. Carry a per-symbol sequence number / timestamp; drop ticks older than lastSeq. Dedupe identical ticks at ingestion so they don't define phantom crossings.

Feed gaps & restarts

On reconnect, the new price may sit far from the stale last price — that interval can legitimately fire crossings. On cold start with no prior, seed last price and don't fire, so we never replay history as live crossings.

▸

The single most important guardrail: seed-don't-fire on first observation. Without it, a restart re-fires every alert already past its target — a mass false-alarm in a finance product, which is a trust incident, not just a bug.

07

Takeaways — what I'd lead with

If I had two minutes to summarise the design on a whiteboard.

  1. "Crosses" means a transition, so the system is stateful — keep the last price per symbol. That's the whole insight.
  2. Partition by symbol so last price + alerts are co-located on one single-writer worker; crossing detection is local and lock-free.
  3. Sorted threshold index turns each tick into an O(log n + k) interval query, not a scan over millions of alerts.
  4. Split the hot match path from slow delivery with a queue; dedupe at delivery with an idempotency key for an exactly-once feel.
  5. Scale partitions and fan-out, never the comparison; conflate hot symbols; rebuild in-memory state from the durable Alert Store on failure.
  6. Seed-don't-fire and hysteresis protect against phantom crossings on restart and flapping on the line — the failure modes that erode user trust.
